# Техническое задание для Codex: доработка анализа отзывов

## Цель

Доработать проект так, чтобы он принимал на вход один JSON-файл с отзывами компании и формировал на выходе два JSON-файла:

1. `review_aspects.json` — результат анализа каждого отзыва: общий sentiment, confidence и список аспектов.
2. `recommendations.json` — агрегированные рекомендации для бизнеса на основе всех отзывов.

Решение должно быть устойчивым к большому количеству отзывов, пустым текстам, разным источникам, разным датам, emoji, переносам строк и опциональным полям.

---

## Входной файл

На вход поступает JSON-файл следующей структуры:

```json
{
  "schema_version": "1.0",
  "analysis_job_id": "28d98c2e-eb09-4556-8b35-0f82bc5f1e09",
  "company_id": "aaaaaaaa-1111-1111-1111-111111111111",
  "reviews": [
    {
      "review_id": 1,
      "text": "Благодарна за качественную работу и ответственное отношение к клиенту!",
      "source": "2gis",
      "date": "2026-03-27T12:30:28.00132+00:00",
      "stars": 5,
      "branch_id": "bbbbbbbb-2222-2222-2222-222222222222"
    }
  ]
}
```

### Обязательные поля верхнего уровня

| Поле | Тип | Обязательность | Описание |
|---|---:|---:|---|
| `schema_version` | string | да | Версия схемы входного файла. На текущем этапе ожидается `"1.0"`. |
| `analysis_job_id` | string | да | UUID задания анализа. |
| `company_id` | string | да | UUID компании. |
| `reviews` | array | да | Массив отзывов. Может быть пустым. |

### Поля объекта `reviews[]`

| Поле | Тип | Обязательность | Описание |
|---|---:|---:|---|
| `review_id` | string / number | да | Идентификатор отзыва. В выходных данных приводить к строке. |
| `text` | string | да | Текст отзыва. Может быть пустой строкой. |
| `source` | string | да | Источник отзыва, например `2gis`, `yandex`, `google`. |
| `date` | string | да | Дата в ISO-8601. |
| `stars` | number | да | Оценка пользователя. Обычно от 1 до 5. |
| `branch_id` | string | да | UUID филиала. |
| `text_language` | string | нет | Язык текста, если источник его передал. |

## Выходной файл 1: `review_aspects.json`

Файл должен содержать массив объектов, по одному объекту на каждый входной отзыв.

### Формат одного элемента

```json
{
  "review_id": "string",
  "text": "string",
  "overall_sentiment": "string",
  "overall_confidence": "number",
  "aspects": [
    {
      "topic": "string",
      "sentiment": "string",
      "confidence": "number",
      "fragment": "string",
      "is_freeform": "boolean"
    }
  ]
}
```

### Требования к полям

| Поле | Тип | Описание |
|---|---:|---|
| `review_id` | string | Идентификатор исходного отзыва, приведённый к строке. |
| `text` | string | Исходный текст отзыва без потери emoji, переносов строк и пунктуации. |
| `overall_sentiment` | string | Общая тональность отзыва. Допустимые значения: `positive`, `negative`, `neutral`, `mixed`, `unknown`. |
| `overall_confidence` | number | Уверенность в общей тональности от `0` до `1`. |
| `aspects` | array | Список найденных аспектов. Может быть пустым, если текст пустой или аспект не удалось выделить. |

### Требования к `aspects[]`

| Поле | Тип | Описание |
|---|---:|---|
| `topic` | string | Тема аспекта. Например: `качество лечения`, `врач`, `администратор`, `цена`, `интерьер`, `ожидание`, `диагностика`, `коммуникация`, `чистота`, `оборудование`. |
| `sentiment` | string | Тональность аспекта. Допустимые значения: `positive`, `negative`, `neutral`, `mixed`, `unknown`. |
| `confidence` | number | Уверенность по аспекту от `0` до `1`. |
| `fragment` | string | Короткий фрагмент исходного текста, подтверждающий аспект. Не придумывать фрагменты, использовать только текст отзыва. |
| `is_freeform` | boolean | `false`, если аспект попал в заранее заданную категорию; `true`, если тема была выделена свободно и не попала в справочник. |

## Выходной файл 2: `recommendations.json`

Файл должен содержать один объект с агрегированными рекомендациями по всем отзывам.

### Формат

```json
{
  "recommendations_count": "number",
  "summary": "string",
  "full_recommendations": [
    {
      "priority": "number",
      "topic": "string",
      "title": "string",
      "body": "string",
      "expected_impact": "string",
      "evidence": [
        "string"
      ]
    }
  ]
}
```

### Требования к полям

| Поле | Тип | Описание |
|---|---:|---|
| `recommendations_count` | number | Количество объектов в `full_recommendations`. |
| `summary` | string | Краткое резюме ключевых выводов по отзывам. |
| `full_recommendations` | array | Список рекомендаций. Может быть пустым, если отзывов нет или данных недостаточно. |

### Требования к `full_recommendations[]`

| Поле | Тип | Описание |
|---|---:|---|
| `priority` | number | Приоритет рекомендации. `1` — самый высокий. |
| `topic` | string | Тема рекомендации. |
| `title` | string | Короткий заголовок рекомендации. |
| `body` | string | Подробное описание, что именно нужно сделать. |
| `expected_impact` | string | Ожидаемый эффект для бизнеса или клиентского опыта. |
| `evidence` | string[] | Список коротких доказательств из отзывов: фрагменты, наблюдения или ссылки на `review_id`. |

## Бизнес-логика анализа

### 1. Общая тональность отзыва

Определять `overall_sentiment` по смыслу текста и, при необходимости, учитывать `stars`.

Рекомендуемая логика:

- `positive` — отзыв явно положительный, клиент доволен.
- `negative` — отзыв явно отрицательный, клиент недоволен.
- `neutral` — отзыв без явной оценки или только фактологическое описание.
- `mixed` — в отзыве есть и сильные положительные, и сильные отрицательные части.
- `unknown` — текст пустой, непонятный или недостаточный для анализа.

Оценка `stars` не должна полностью заменять анализ текста. Например, отзыв на 5 звёзд может содержать важную претензию, а отзыв на 2 звезды может содержать положительную оценку отдельного сотрудника.

### 2. Аспектный анализ

Нужно выделять отдельные аспекты внутри каждого отзыва.

Примеры аспектов:

| Тема | Примеры сигналов |
|---|---|
| `качество лечения` | качественно, безболезненно, помогли, зуб спасли, результатом доволен |
| `врач` | внимательный врач, профессионал, объяснил план, не хотел уделять внимание |
| `администратор` | ресепшн, администратор, запись, WhatsApp, игнорировали |
| `сервис` | встретили, предложили чай/кофе, отношение, забота |
| `цена` | дорого, недорого, приемлемые цены, акция |
| `ожидание` | ждал час, приняли вовремя, по записи |
| `интерьер` | уютно, современно, красиво, не располагает |
| `чистота` | пыль, чисто, стерильно |
| `оборудование` | микроскоп, КТ, 3D-сканер, аксиограф, современное оборудование |
| `диагностика` | снимки, КТ, диагностика, план лечения |
| `коммуникация` | объяснили, ответили на вопросы, не дали информацию |
| `детский приём` | ребёнок, дочь, сын, ортодонт, не испугался |
| `ортодонтия` | брекеты, элайнеры, прикус, пластины |
| `имплантация` | имплант, протезирование на имплантах, костная пластика |
| `протезирование` | коронки, виниры, протезы, all-on-4, all-on-6 |
| `ВНЧС / гнатология` | сустав, щелчки, челюсть, капа, сплинт |

Если тема не попадает в известный список, разрешено создать новую тему и поставить `is_freeform: true`.

### 3. Фрагменты-доказательства

Для каждого аспекта нужно сохранять фрагмент из исходного отзыва.

Правила:

- `fragment` должен быть дословной частью `text`.
- Не генерировать фрагмент, которого нет в отзыве.
- Фрагмент должен быть коротким, но достаточным для проверки аспекта.
- Если аспект основан на общем смысле длинного текста, выбрать наиболее показательный фрагмент.
- Для пустого текста `aspects` должен быть пустым массивом.

### 4. Confidence

Все confidence должны быть числами от `0` до `1`.

Рекомендуемые значения:

- `0.9–1.0` — явная оценка, сильный сигнал.
- `0.7–0.89` — понятная оценка, но есть небольшой контекстный риск.
- `0.5–0.69` — слабый или косвенный сигнал.
- `<0.5` — неуверенный вывод, лучше использовать `unknown` или не добавлять аспект.

## Логика формирования рекомендаций

Рекомендации должны строиться на агрегированных паттернах по всем отзывам, а не на единичных случайных упоминаниях, кроме случаев с высоким риском для бизнеса.

### Нужно учитывать

1. Частотность темы.
2. Долю негативных упоминаний.
3. Свежесть отзывов.
4. Потенциальный бизнес-ущерб.
5. Повторяемость проблемы в разных источниках.
6. Серьёзность проблемы: безопасность, медицинское качество, коммуникация, запись, администраторы, навязывание услуг, ожидание, чистота.

### Примеры рекомендаций для подобных данных

- Улучшить коммуникацию администраторов и обработку сообщений в WhatsApp.
- Ввести стандарт объяснения дальнейшей диагностики после консультации.
- Усилить контроль чистоты кабинетов и зон ожидания.
- Отслеживать негативные отзывы по конкретным врачам и проводить внутренний разбор.
- Использовать сильные стороны в маркетинге: современное оборудование, лечение под микроскопом, имплантация, гнатология, комплексные планы лечения.
- Выделить детский приём и ортодонтию как отдельные сильные направления.
- Проверять отзывы на признаки накрутки или шаблонности, если много похожих текстов в короткий период.

### Приоритеты

`priority` должен начинаться с `1`.

Рекомендуемая шкала:

- `1` — критично: влияет на доверие, безопасность, повторные визиты или репутацию.
- `2` — важно: заметно влияет на клиентский опыт и конверсию.
- `3` — полезно: улучшает маркетинг, удержание или операционные процессы.

## Обработка ошибок и edge cases

### Пустой массив отзывов

Если `reviews` пустой:

`review_aspects.json`:

```json
[]
```

`recommendations.json`:

```json
{
  "recommendations_count": 0,
  "summary": "Недостаточно данных для формирования рекомендаций: входной файл не содержит отзывов.",
  "full_recommendations": []
}
```

### Пустой текст отзыва

Если `text` пустой:

```json
{
  "review_id": "225",
  "text": "",
  "overall_sentiment": "unknown",
  "overall_confidence": 0,
  "aspects": []
}
```

### Некорректный JSON

При некорректном JSON приложение должно завершаться с понятной ошибкой и не создавать частично заполненные выходные файлы.

### Отсутствующие обязательные поля

Если отсутствует обязательное поле верхнего уровня или обязательное поле отзыва, нужно:

1. Вернуть понятную ошибку в логах или stderr.
2. Не падать с неинформативным stack trace.
3. Не создавать некорректные выходные файлы.

### Нестандартные типы

- `review_id` на выходе всегда приводить к строке.
- `stars` может быть числом или строкой, если входной источник передал неконсистентные данные. По возможности приводить к числу, но не ломать анализ текста.
- Дополнительные поля входного отзыва не должны ломать обработку.

## Требования к CLI / запуску

Если в проекте уже есть CLI, доработать существующий интерфейс. Если CLI нет, добавить минимальный запуск:

```bash
python -m app.analyze_reviews --input input.json --out-dir ./out
```

После выполнения в директории `out` должны появиться:

```text
review_aspects.json
recommendations.json
```

Допустимо использовать другие имена модулей, если структура проекта уже отличается, но интерфейс должен быть документирован в README или help-команде.

## Требования к качеству JSON

1. Выходные файлы должны быть валидным JSON.
2. Кодировка — UTF-8.
3. Не экранировать кириллицу без необходимости: желательно `ensure_ascii=false`.
4. JSON должен быть человекочитаемым: `indent=2`.
5. Не добавлять markdown, комментарии или поясняющий текст внутрь JSON.
6. Все числа confidence должны быть числами, а не строками.
7. Все boolean-поля должны быть boolean, а не строками `"true"` / `"false"`.

---

## Требования к тестам

Добавить или обновить тесты.

Минимальный набор тестов:

1. Успешная обработка валидного файла с несколькими отзывами.
2. Корректная обработка пустого массива `reviews`.
3. Корректная обработка пустого текста отзыва.
4. `review_id` приводится к строке.
5. На выходе создаются ровно два JSON-файла.
6. `recommendations_count` равен длине `full_recommendations`.
7. Все `overall_confidence` и `aspects[].confidence` находятся в диапазоне `0..1`.
8. `fragment` является подстрокой исходного `text`, если `fragment` не пустой.
9. Ошибка при некорректном JSON.
10. Ошибка при отсутствии обязательных полей.

---

## Критерии приёмки

Задача считается выполненной, если:

1. Проект принимает входной JSON указанного формата.
2. Проект создаёт два выходных файла:
   - `review_aspects.json`
   - `recommendations.json`
3. Оба файла полностью соответствуют заявленным JSON-схемам.
4. Анализ не падает на emoji, переносах строк, длинных отзывах и пустых текстах.
5. Для каждого входного отзыва есть один объект в `review_aspects.json`.
6. Рекомендации основаны на агрегированных данных по отзывам.
7. Все тесты проходят.
8. В README или help-команде описан способ запуска.
9. Код не содержит захардкоженных данных конкретной компании, конкретных `review_id` или конкретного примера.
10. Решение работает не только на предоставленном примере, но и на любом JSON той же схемы.

---

## Нежелательное поведение

Не допускается:

1. Генерировать выходные данные в markdown вместо JSON.
2. Терять отзывы из входного массива.
3. Менять исходный текст отзыва в поле `text`.
4. Придумывать `fragment`, которого нет в исходном отзыве.
5. Использовать только `stars` без анализа текста.
6. Игнорировать негативные аспекты в положительных отзывах.
7. Создавать рекомендации на основе одного слабого сигнала, если это не критичный риск.
8. Падать на пустых строках, emoji или дополнительных полях.
9. Возвращать confidence строкой.
10. Возвращать boolean строкой.

## Примечания для реализации

- Сохранять исходную сортировку отзывов в `review_aspects.json`.
- Рекомендации сортировать по `priority` по возрастанию.
- При одинаковом приоритете сначала выводить рекомендации с большим количеством evidence.
- Желательно логировать количество обработанных отзывов, количество пустых текстов и количество созданных рекомендаций.
- Если используется LLM, промпт должен требовать строго валидный JSON без markdown-обёртки.
- Если используется локальная эвристика, она должна быть покрыта тестами и не быть завязанной только на конкретный пример входных данных.
