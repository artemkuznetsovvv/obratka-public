# CLAUDE-dashboards.md — гид для агента по реализации дашборда

Этот файл — точка входа для агента, который будет реализовывать дашборд анализа
**итеративно, по одной метрике за сессию**. Юзер шлёт описание одной метрики
(или подзадачи зонтичной задачи) — агент собирает контекст из этого файла,
задаёт уточняющие вопросы и реализует.

> Прежде чем что-то писать — прочитай этот файл целиком + указанные ниже
> source-of-truth документы. Не угадывай, как считать метрику — открой
> спецификацию и сопоставь с реальной схемой БД.

## 0. Главный принцип — следуем ADR (но это **draft**, реальность — в коде)

ADR (`ADR-obratka-app/`) — **источник истины по архитектурным решениям**:
направление, разделение модулей, изоляция, паттерны интеграции. При любом
расхождении в этом — **побеждает ADR** (правило из root
[`../CLAUDE.md`](../CLAUDE.md)).

**НО:** конкретные **схемы таблиц / DTO / поля** в тексте ADR — это
**предварительный proposal**. Реальная схема может уже расходиться (типы
колонок, имена полей, опц. nullable). Перед тем как опираться на конкретное
поле — **сверься с фактическим состоянием**:

- Entity-классы и `WebApiDbContext` / `ProcessingDbContext`
- Текущая миграция (последняя в `Migrations/`)
- Если сомневаешься в реальных данных — **попроси юзера выгрузить SQL**.
  Он подключится к нужной БД (`webapi_db` или `processing_db`) и пришлёт
  результаты. Используй это для верификации схемы и тест-кейсов.

### Что из ADR соблюдаем для дашбордов твёрдо

- **ADR-003** — общий паттерн: pre-computed snapshots, 4 таблицы
  (`analysis_snapshots` / `metric_timeseries` / `branch_snapshots` /
  `topic_stats`), counts а не ratios. Точные DDL — это draft, реализация
  может расходиться по мере того как поймём реальные edge-кейсы.
- **ADR-011** — Analytics-модуль **физически читает `processing_db`**
  (где живут `reviews` и `review_llm_results`) через отдельного PG-пользователя
  `analytics_reader` с правами `SELECT` только на 3 таблицы. В коде это
  второй EF DbContext `ProcessingReadContext` (read-only, `SaveChanges` бросает
  исключение). Это **сознательный MVP trade-off** (schema coupling), описан в
  ADR-011 §«MVP trade-off». Когда модуль будут выносить в отдельный сервис —
  переключатся на API/события вместо прямого читания. До тех пор — **не
  бойся читать `processing_db`**, это норма для Analytics-модуля.
- **ADR-002 / ADR-004** — схема `reviews` / `review_llm_results` и формат LLM
  schema 2.0 (`overall_sentiment`/`aspects`/`summary`/`full_recommendations`)
  — фактическая реализация в PG, можно опираться, но всё равно сверять с
  кодом перед чтением.

### Что делать при конфликте

- Спека PDF противоречит ADR-направлению (например, «считать на лету» вместо
  pre-computed) → **не делай молча**. Через `AskUserQuestion`: «спека
  предлагает X, ADR-003 направление Z — идём ADR? Или нужен bump ADR?».
- Текст ADR противоречит коду (конкретные поля/типы) → **трактуй ADR как
  draft**. Действуй по коду, отметь расхождение юзеру, при сильном
  расхождении предложи обновить ADR отдельным коммитом в `ADR-obratka-app/`.
- Если в ADR описано **направление**, для которого нужна таблица/поле — назови
  её как в ADR (минимизируем расхождения), но точную форму выбирай под код.

## 1. Что мы делаем

Финальная точка пользовательского пути:
**компания → филиалы → период → запуск → ДАШБОРД**.

Дашборд показывает результаты LLM-анализа отзывов — KPI, динамику, темы,
болевые точки, рекомендации. Состоит из **10 метрик-карточек** + фильтров +
шапки с параметрами анализа. Логика секций зависит от количества выбранных
филиалов (1 / 2 / 3+).

## 3. Что уже реализовано (контекст для агента)

К моменту начала работы над дашбордом сделано:

### Воронка запуска анализа
- Мастер на 3 шага: компания+период+источники → группировка филиалов → сводка → запуск
- Auto-grouping карточек разных источников в `LogicalBranch` (физический филиал)
- Реальный запуск через PG: `POST /api/analyses/start` → `StartAnalysisCommand` → весь pipeline до `completed`
- В PG `reviews.branch_id` = `LogicalBranch.Id` (физический филиал), `reviews.source` — отдельная dimension. **Это ключевое:** разрезы дашборда по филиалу и по источнику работают через две независимые колонки без cross-DB JOIN'ов.

### Страница «История анализов»
- `/history` — список джобов с поллингом
- `/history/:jobId` — детальная: stepper 1→4, круговой прогресс, список 5 LLM-стадий, per-source breakdown, collapsible «Параметры анализа», блок «Результаты» с per-branch counts
- Сейчас в Результатах есть только заглушка «Полный дашборд и список рекомендаций — в следующих обновлениях»
- **Сюда (или на отдельный URL) и будем встраивать дашборд**

### Чего НЕТ (важно для дашбордов)
- `Obratka.Modules.Analytics` — только skeleton, без таблиц, consumer'ов, endpoint'ов
- `ProcessingReadContext` (read-only DbContext к `processing_db` через `analytics_reader`) — не создан
- MassTransit subscription на `AnalysisCompletedEvent` от PG — не подписаны
- Таблицы `analysis_snapshots`, `metric_timeseries`, `branch_snapshots`, `topic_stats` — нет миграций
- `AggregatesReadyEvent` от Web API → PG — не публикуется

Это и есть зона работы дашбордов. Дорожная карта ниже.

### 4.2. Поведение фильтров

- Фильтры влияют на **все** метрики базового и расширенного слоёв.
- **Исключения:** метрика 4 («Свежий пульс» — окно 30 дней) и метрика 7
  («Новые отзывы за период» — переключатель внутри карточки) **не реагируют**
  на фильтр периода.
- Изменение любого фильтра → пересчёт всех метрик.

## 5. Архитектура реализации (обязательно следовать)

### 5.1. Pre-computed > runtime SQL по сырым данным

ADR-003 явно фиксирует: dashboard читает **готовые числа** из таблиц
`analysis_snapshots` / `metric_timeseries` / `branch_snapshots` / `topic_stats`.
Никаких `GROUP BY` по `reviews` × `review_llm_results` в рантайме дашборда.

Пересчёт — **один раз** по завершении анализа (consumer на
`AnalysisCompletedEvent`). Реализация — в `Obratka.Modules.Analytics`.

### 5.2. Исключение — список отзывов в блоке «Топ примеров»

ADR-003 разрешает **одно** rt-обращение к raw: выборка топ-отзывов с
фильтрами тональности/звёзд. Запрос лёгкий (индекс по `analysis_job_id`,
LIMIT 5), бежит через `ProcessingReadContext` (read-only).

### 5.3. Где какие данные

| Запрос | Источник |
|---|---|
| KPI-карточки на конкретный анализ (O1-O3, 1-3, 6) | `analysis_snapshots` + `branch_snapshots` (готовые числа) |
| Тренды (динамика по неделям) | `metric_timeseries` |
| Топ-темы (метрика 5) | `topic_stats` |
| Свежий пульс (метрика 4, окно 30д) | `metric_timeseries` за последние 4 недели |
| Новые отзывы за период (метрика 7) | `metric_timeseries` с переключателем гранулярности |
| Список «топ позитивных / негативных» | `reviews JOIN review_llm_results WHERE analysis_job_id=? AND overall_sentiment=? LIMIT 5` (raw, через `ProcessingReadContext`) |
| Фильтр по теме | Применяется к pre-computed либо через `topic_stats.topic` |

### 5.4. Counts vs ratios

Все агрегаты хранят **counts** (числа отзывов в каждом sentiment-bucket'е), **не проценты**.
Это позволяет суммировать недельные строки в месяц/квартал без потери точности.
NPS вычисляется в application layer из counts.

### 5.5. Слои изоляции

- Analytics-модуль читает `processing_db` через **`analytics_reader`** —
  отдельный PostgreSQL-пользователь с `SELECT` только на 3 таблицы
  (`reviews`, `review_llm_results`, `analysis_jobs`). Без `INSERT/UPDATE/DELETE`.
- `ProcessingReadContext` — DbContext, в котором `SaveChanges` бросает исключение.
- Реальное соединение к `processing_db` — отдельная connection string
  `ConnectionStrings__ProcessingDb` (которой сейчас в .env Web API нет — добавить
  при подключении Analytics).

## 6. Дорожная карта (последовательность задач)

Чтобы реализация шла предсказуемо, скорее всего пойдём так (но юзер задаёт
порядок — он может прислать любую метрику в любой момент):

### Фаза 0 — инфраструктура Analytics-модуля (один раз)

1. Создать `ProcessingReadContext` + регистрация второго connection string
2. Подписка на `AnalysisCompletedEvent` (`MassTransit` consumer)
3. Миграция: таблицы `analysis_snapshots`, `metric_timeseries`,
   `branch_snapshots`, `topic_stats` (по ADR-003)
4. Skeleton `AggregatesComputer` — пустой compute, публикация `AggregatesReadyEvent`
5. Endpoint `GET /api/analyses/{jobId}/dashboard` — пустой return
6. Frontend: страница `/history/:jobId/dashboard` (или замена placeholder'а
   в `HistoryDetailPage`) с шапкой и контейнером для карточек

### Фаза 1 — главный экран (зонтичная задача)

7. Шапка + фильтры (UI), без подключения метрик
8. Логика «1 / 2 / 3+ филиалов»
9. Кнопки «PDF» / «Включить мониторинг» — пока заглушки

### Фаза 2 — метрики по одной

10. Каждая метрика отдельно: спека → решение по compute → backend (агрегат +
    endpoint) → frontend (карточка) → подключение к фильтрам → проверка на
    реальных данных через query к БД.

## 7. Процесс работы над **одной метрикой**

Когда юзер пришлёт задачу типа «Реализуй метрику X», действуй так:

### Шаг 1. Контекст

- Прочитай `CLAUDE-dashboards.md` (этот файл) — освежи в памяти архитектурные ограничения
- Прочитай нужные страницы спецификации (`Web/metrics_info/Обратка_Метрики_Спецификация.pdf`)
- Загляни в [ADR-003](../ADR-obratka-app/ADR%20по%20компонентам%20системы%20+%20описание%20системы/details/ADR-003-analytics-database.md) на предмет описанного **направления** (помни §0: точные DDL — draft)
- **Проверь реальную схему** в Entity-классах / последних миграциях — на тебе она актуальна, не на ADR
- Если есть макет для этой страницы / карточки в `Web/design/_N/` — посмотри `screen.png` и `code.html` (см. §2.3)
- Если фаза 0 (инфраструктура) ещё не сделана — начни с неё, не пытайся посчитать метрику без `analysis_snapshots`

### Шаг 2. Анализ — задай вопросы юзеру

Через `AskUserQuestion`:

- **Как считается?** Подтверди формулу из спеки своими словами, спроси согласие
- **Гранулярность данных:** snapshot per-job? timeseries per-week? per-branch? per-source?
- **Фильтры:** какие из 6 фильтров влияют, какие игнорируются (см. §4.2)
- **Edge-кейсы:**
  - Нет отзывов за период → 0? null? «Данных пока нет»?
  - Меньше N отзывов → выводить с warning «недостаточно данных»?
  - Отсутствует sentiment_confidence (LLM не уверен) → исключать из агрегата?
- **Визуализация:** число, прогресс-бар, donut, line chart, bar chart? Если есть макет в `Web/design/_N/` — опирайся на него; если нет — спроси юзера / предложи сам по дизайн-системе проекта
- **Можно ли проверить на реальных данных?** Юзер выполняет SQL-запросы к `webapi_db` или `processing_db` по твоей просьбе — формулируй запрос конкретно (имена таблиц, фильтры, jobId), проси прислать результат. Это и проверка реальной схемы (типы / nullable), и сверка значений метрики

### Шаг 3. Решение по compute

- **Pre-computed snapshot** (на `AnalysisCompletedEvent`) → строка/несколько строк в `analysis_snapshots` / `branch_snapshots` / `topic_stats`
- **Pre-computed timeseries** (там же) → upsert в `metric_timeseries` per (company × week × source)
- **Runtime SQL** → только если это «топ-N примеров» (ADR-003 §«Фильтры… raw»)

Если не уверен, какая модель — спроси юзера.

## 9. Подсказки агенту по типичным граблям

- **Не считай в рантайме по `reviews`** — нарушение ADR-003. Если соблазн — спроси юзера, не сделать ли pre-compute
- **`branch_id` в `reviews`** = `LogicalBranch.Id` (физический). Имена/адреса филиалов — в `webapi_db.LogicalBranches`, JOIN на стороне Web API (Analytics-модуль внутри неё, доступ есть)
- **Источник** — отдельная колонка `reviews.source` (`'2gis'` / `'yandex'` / `'google'`). Не пытайся вытащить из других полей
- **Sentiment в schema 2.0** — русские строки `'позитивный' / 'негативный' / 'нейтральный'` (см. ADR-004). НЕ `positive/negative/neutral`
- **Aspects** — JSONB-массив объектов `{topic, sentiment, confidence, fragment, is_freeform}`. Один topic может встречаться несколько раз с разными sentiment'ами в одном отзыве
- **GIN-индекс** на `review_llm_results.aspects` уже есть (ADR-002). Для фильтрации по теме используй `aspects @> '[{"topic":"..."}]'`
- **NPS** = (`позитивный` + `очень позитивный`) − (`негативный` + `очень негативный`)) / total × 100. В schema 2.0 нет градации «очень» — sentiment-bucket'ов всего три, формула NPS адаптируется (уточни у юзера)
- **`Company.draftPeriodFrom/To`** — это «настройки следующего анализа», НЕ период каждого конкретного job-а. Период джоба сейчас нигде в БД не лежит, см. [`processing-gateway-todo.md` #1](../Processing-Gateway/processing-gateway-todo.md). Если метрика опирается на период — учитывай этот caveat (либо подталкивай к реализации того TODO)

## 10. Если что-то не сходится — что делать

- **Спека PDF противоречит ADR-003** → дефолт = **побеждает ADR** (§0). Остановись, опиши конфликт явно через `AskUserQuestion`: «спека требует X, ADR-003 §Y описывает Z, по правилу root CLAUDE.md побеждает ADR — подтверждаешь идти путём ADR? Или хочешь сделать bump ADR-003?». Без явного согласия юзера НЕ нарушай ADR
- Не хватает данных в LLM-выхлопе → подними вопрос. Возможно, в `output_summary.json` есть нужное поле и нужно расширить consumer
- Реальные данные не сходятся с расчётом → попроси юзера прислать SQL-выгрузку для конкретного `analysis_job_id`, диагностируй на ней
- Подзадача из ТЗ не покрыта спецификацией → **остановись и спроси**, как считать
- В ADR-003 описано поле, которого нет в коде → реализуй **именно как в ADR**, не придумывай параллельную колонку
